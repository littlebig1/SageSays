# Documentation Rules - Quick Reference

> **This file explains the documentation structure. Do not create additional docs!**

## The 5+1 Rule

We maintain exactly **5 files in `/docs`** + **1 root README**:

```
/
├── README.md                    # Setup guide for new devs
└── docs/
    ├── ARCHITECTURE.md          # How the system works
    ├── ROADMAP.md               # Future plans
    ├── SESSION_NOTES.md         # What we built
    ├── CONTROL_DB_SCHEMA.md     # Database schema
    └── DEVELOPMENT.md           # Contributing guide
```

## Decision Tree: Where Does My Content Go?

```
Start here
│
├─ "How does feature X work?" → ARCHITECTURE.md (add section)
├─ "We should build Y" → ROADMAP.md (add to phase)
├─ "We just built Z" → SESSION_NOTES.md (add at top)
├─ "How do I set up?" → README.md (keep brief)
├─ "How do I contribute?" → DEVELOPMENT.md (add pattern)
└─ "What's the schema?" → CONTROL_DB_SCHEMA.md (update)
```

## ❌ Never Create

- Feature-specific docs (e.g., `SMART_MODE.md`)
- Phase summaries (e.g., `PHASE2_SUMMARY.md`)
- Index files (e.g., `docs/README.md`)
- Duplicate guides

**Why?** Use sections in ARCHITECTURE.md instead.

## ✅ Always Update

- **Every session**: `SESSION_NOTES.md` (add at top)
- **New feature**: `ARCHITECTURE.md` (add section)
- **New plan**: `ROADMAP.md` (update phase)
- **Schema change**: `CONTROL_DB_SCHEMA.md`
- **New pattern**: `DEVELOPMENT.md`

## Pre-Commit Checklist

- [ ] No new files in `/docs` (only 5 allowed)
- [ ] `SESSION_NOTES.md` updated
- [ ] Feature docs added to `ARCHITECTURE.md` if needed
- [ ] Schema docs updated if changed

## Enforcement

This structure is enforced by:
1. This file (`.DOCUMENTATION_RULES.md`)
